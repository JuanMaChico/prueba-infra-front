Beneficios de un Monorepo React con TanStack

Cómo Vite, TanStack Router/Query/Table y un SDK Axios compartido transforman un dashboard de e-commerce en un sistema escalable, tipado y rápido de mantener.

Por Juan Carlos Donis · Stack: Vite + React + pnpm · 15 slides
01 · Contexto

El caos del frontend tradicional

Los dashboards monolíticos acumulan problemas que frenan al equipo y degradan la experiencia.

Lógica de red dispersa

Cada componente llama a fetch o axios por su cuenta. URLs, headers y manejo de errores duplicados en 30+ archivos.

Estados y caché inventados a mano

Loading, error, refetch, invalidación… código a medida que se rompe en cada feature nueva.

Tipos any por todos lados

Sin contrato entre la API y la UI. Los bugs llegan a producción y el refactor se vuelve impagable.

Bundle gigante y builds lentos

Todo se importa en el primer paint. La tabla de 5.000 filas bloquea el navegador.

03 · Stack

Las herramientas, en su rol

Cada pieza resuelve un problema concreto. Ninguna hace trabajo ajeno.

⚡ Vite + React

HMR nativo, build ESM, dev server listo en 300 ms. La base que nunca estorba.

📦 pnpm Workspaces

~70% menos node_modules, linking simbólico, lockfile determinista para CI.

🧭 TanStack Router

File-based, type-safe, beforeLoad para guardias. Cero acoplamiento a la API.

🔄 TanStack Query

Caché, retry, invalidación y refetch declarativos. El estado del servidor, resuelto.

📊 TanStack Table

Headless, virtualizable, ~30 ms por frame con 5.000 filas. Sin librería pesada de UI.

🎨 TailwindCSS

Utility-first + JIT. Estilos coherentes, bundle pequeño, sin CSS muerto.

Beneficio · 01

Separación clara de responsabilidades

Tres capas que no se mezclan. Cambias una sin romper las otras dos.

Presentación

React + Tailwind

Componentes tontos. Reciben datos vía hooks, renderizan, manejan interacción local.

const { data: orders } = useOrdersQuery();
Estado de servidor

TanStack Query

Caché, reintentos, deduplicación, refetch en foco. Los hooks envuelven las funciones del SDK.

useQuery(['orders'], getOrders, { staleTime: 60_000 })
Transporte

SDK Axios

Una sola instancia, interceptores centralizados, adaptadores por dominio. Es el único que sabe de HTTP.

export const getOrders = () => axiosInstance.get('/orders');

"La UI solo sabe que llama a una función que devuelve una promesa."

Beneficio · 02

SDK reutilizable y contratos tipados

Un paquete @api-client que cualquier app del monorepo puede consumir. Mismos tipos, misma baseURL, mismas reglas de auth.

🔌 Una sola instancia Axios

Timeout global, headers por defecto, interceptores de request/response. La auth vive aquí.

axiosInstance.interceptors.request.use(cfg => { const token = localStorage.getItem('access_token'); if (token) cfg.headers.Authorization = `Bearer ${token}`; return cfg; });

🧩 Adaptadores por dominio

Una carpeta por dominio: orders/, products/, users/. Cada uno expone funciones puras.

export const getOrders: () => Promise<Order[]>; export const createOrder: (n: NewOrder) => Promise<Order>; export const updateOrder: (id: string, p: Partial<Order>) => ...

El resultado: el backend cambia y tú solo tocas un paquete

Si el contrato de la API evoluciona, se actualizan los tipos del SDK y TypeScript marca el camino en cada componente. Cero any propagados, cero errores silenciosos.

Beneficio · 03

Data fetching declarativo

Olvídate de useEffect + setLoading + setError. TanStack Query te devuelve un objeto rico y predecible.

Hooks limpios y testeables

Cada feature tiene su hook. La UI solo lee isLoading, error y data.

export const useOrdersQuery = () => useQuery(['orders'], getOrders, { staleTime: 60_000, refetchOnWindowFocus: false, retry: 2, });

Mutaciones que invalidan solas

onSuccess invalida la clave correspondiente y la UI se refresca sin reescribir nada.

const qc = useQueryClient(); return useMutation(createOrder, { onSuccess: () => qc.invalidateQueries(['orders']), });

Beneficio · 04

Routing file-based y code splitting gratis

Una carpeta = una ruta. Vite genera el chunk. El usuario descarga solo lo que visita.

// apps/web/src/routes/orders.tsx import { Route } from '@tanstack/react-router'; import { OrdersTable } from '../components/OrdersTable'; export const ordersRoute = new Route({ getParentRoute: () => rootRoute, path: '/orders', beforeLoad: async ({ context }) => { if (!context.auth.user) throw new Error('Unauthenticated'); }, component: () => <RequireAuth><OrdersTable /></RequireAuth>, });

📁 File-based

Creas products.tsx y la ruta /products existe. Sin config central.

🪶 Lazy por defecto

Cada ruta es su propio chunk. El bundle de Orders pesa ~45 KB gzipped.

🛡️ Guardias declarativas

beforeLoad corta antes de renderizar. Cero parpadeo de UI no autenticada.

Beneficio · 05

Beneficio · 07

Testing robusto sin fricción

Dos niveles: el SDK se prueba con mocks de red, los hooks se prueban con MSW + React Query. La cobertura dicta el merge.

🧪 Test del SDK

MSW intercepta fetch a nivel de Node, así que el test es indistinguible del runtime real.

test('getOrders returns an array', async () => { const orders = await getOrders(); expect(orders).toHaveLength(1); expect(orders[0].id).toBe('1'); });

🪝 Test de hooks

Wrapper con QueryClientProvider + renderHook + waitFor.

const { result } = renderHook(() => useOrdersQuery(), { wrapper }); await waitFor(() => expect(result.current.isSuccess).toBeTruthy() );

Beneficio · 08

Optimizaciones listas para producción

El setup ya está alineado con las mejores prácticas. Solo enciendes las que necesitas.

Lazy-load de rutas

lazy: () => import('./orders') en routes/index.ts. Vite crea el chunk; el usuario no lo descarga si no lo visita.

Prefetch en hover

router.prefetch('/orders') en los links. El primer clic se siente instantáneo.

Cache-busting automático

Vite inyecta hash de contenido en el nombre de los assets. Cache-Control: max-age=31536000 sin miedo.

Tailwind en modo JIT

Solo se incluyen las clases que usas. Bundle CSS drásticamente más liviano.

Deploy a CDN

pnpm --filter web build → apps/web/dist listo para Netlify, Vercel o Cloudflare Pages.

04 · Trampas a evitar

Errores comunes al implementar

La mayoría son fáciles de evitar si los tienes en mente desde el día uno.

Sí hacer

  • Una sola instancia Axios exportada y reutilizada.
  • Tipos del SDK importados en cada hook.
  • invalidateQueries en cada onSuccess de mutación.
  • Guardias en beforeLoad, no dentro del componente.
  • staleTime realista (30 s – 2 min) y refetchOnWindowFocus.
  • Solo pnpm, nunca npm install.

No hacer

  • Importar axios directamente desde un componente.
  • Crear axios.create() dentro de cada adaptador.
  • Tipos any en las respuestas del SDK.
  • staleTime: Infinity y caché indefinido.
  • Lógica de auth duplicada en cada componente.
  • Olvidar las rutas content de Tailwind (CSS no purgado).
